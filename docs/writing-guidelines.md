# Writing Guidelines

## Goal

Write like someone who understands both the mathematics and the student failure
modes, and is now turning that understanding into a dependable course resource.

The notes should feel:

- mathematically serious
- explicit about proof expectations
- calm and readable
- formal enough to feel trustworthy
- careful without sounding sterile
- usable as a primary study resource, not just a companion

## What these notes are trying to do

Most students do not only need more examples. They need help with:

- reading the shape of a statement
- knowing which definition matters
- seeing what kind of proof the problem is asking for
- understanding when the course expects formal rule citation and when it does not

That means each note should teach both:

- the topic itself
- the local writing habits of that topic

## Good note pattern

1. Start with the distinction that matters.
2. Say what kind of object or statement is on the page.
3. If a theorem is involved, separate hypotheses from conclusion.
4. Expand the exact definition that drives the reasoning.
5. Show what a proof here usually has to do.
6. Include a small number of carefully chosen examples.
7. If the proof needs real derivation, write it out rather than summarizing it.
8. End by summarizing the local takeaway or pointing cleanly to the next skill.

## Tone

- direct
- compact
- specific
- editorially steady
- more like staff-written course notes than guided commentary

The writing should not sound like:

- copied lecture notes
- a textbook
- a polished motivational essay
- a generic AI explanation
- a private study diary
- notes about the notes themselves

## Strong sentence-level defaults

- Prefer "what is this statement really asking for?" over broad motivation.
- Prefer "this is the definition doing the work" over vague summary.
- Prefer "the hard part is..." over pretending a step is automatic.
- Prefer a short proof skeleton when that helps more than another paragraph.
- Prefer a functional heading over a clever heading.
- Prefer the formal definition in math once the object has been introduced in words.
- Prefer a case split when the statement naturally separates into cases.
- Prefer explicit notation over informal ellipsis inside a formal proof.

## Things to avoid

- throat-clearing openings
- generic transitions
- inflated language
- proving too many small claims in one paragraph
- examples that do not expose the actual proof move
- repeated reminders that the note is only a draft or companion
- filler sentences like "this vocabulary is not cosmetic" when the mathematical consequence can be stated directly
- "in this course" filler when the claim is really just the definition or proof rule itself
- large or decorative visuals that do not clarify a specific definition

## Openings

Openings should vary with the topic, but the page should still read like a
course note, not a personal essay. A strong opening usually does one of these:

- states the distinction the unit depends on
- identifies the kind of sentence or object being studied
- names the proof habit the section is training
- clarifies a common misclassification students make

Once the opening has done its work, move quickly into the mathematics.

## Visuals

Use visuals sparingly.

Good reasons to add a visual:

- to show a walk that repeats a vertex
- to show graph features excluded by a definition
- to distinguish two closely related local objects

When a visual is used, it should be:

- minimal
- clean
- textbook-like
- subordinate to the mathematics
