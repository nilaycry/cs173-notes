# Note Standards

## Purpose

This document defines the local structure for the main page types in the site.

The goal is consistency without flattening every note into the same voice.

## Shared structural rules

Every note should:

- use functional headings
- define the local object of study early
- keep paragraphs short
- make the relevant proof habit visible
- use formal notation once the object is introduced
- end with a concise transition or takeaway when it helps

Do not end notes with apology, diary-like hedging, or "this note is not doing"
language unless that boundary is mathematically necessary.

## Concept Note

Use this when the page teaches a piece of mathematical content.

Typical structure:

1. Core distinction or classification
2. Definition or statement form
3. Example or counterexample
4. Proof habit or interpretive warning
5. Transition to the next concept

For graph-theory concept notes, also prefer:

- explicit function/set notation for walks, paths, neighborhoods, and degree
- compact lists of immediate consequences of a definition
- minimal examples with concrete vertices when a repeated-node or excluded-case issue matters

Strong heading patterns:

- `Declarative Sentences`
- `Truth Value`
- `Logical Equivalence`
- `Implication and Premise`

## Proof-Writing Note

Use this when the page teaches how solutions are supposed to be written in a
particular unit.

Typical structure:

1. What this unit is asking for
2. Common proof tasks
3. Expectations for a complete writeup
4. Frequent errors
5. What changes in the next unit

The page should read like guidance a course staff member would be comfortable
handing to students.

## Problem Guide

Use this when the page explains a worksheet, problem set, homework, or practice
exam.

Typical structure:

1. What the set is testing
2. Representative problem types
3. What the writeups need to do
4. Common errors
5. How the set connects to the next topic

The guide should not become a raw answer key.

If a single problem introduces a central proof technique, it is acceptable for
the guide to include one full model derivation in the same level of formality as
the course.

## Review Note

Use this when the page is meant for midterm or final preparation.

Typical structure:

1. Core definitions or statements to know
2. Typical proof forms
3. Common confusions
4. Practice prompts or classifications
5. What to check before moving on

## Heading conventions

Prefer headings that help scanning and review.

Prefer:

- `Declarative Sentences`
- `Criteria for Truth Value`
- `Expectations for a Complete Writeup`

Avoid:

- `A Better Way To Hold the Idea`
- `Where the Confusion Really Comes From`
- `What This Note Is Not Doing Yet`

## Ending conventions

Good endings usually do one of the following:

- summarize the mathematical takeaway
- identify the proof habit the reader should now have
- point directly to the next concept

Avoid endings that sound provisional for their own sake.

## Things to cut on revision

Remove lines that mainly do one of the following:

- explain that the note has a philosophy
- comment on the notes as notes
- restate a definition without adding mathematical content
- say that terminology "matters" without specifying the theorem or proof consequence
